# 分布式训练基础

## 概述

分布式训练是将大语言模型（LLM）的训练任务拆分到多台机器、多张 GPU 上协同完成的技术。随着模型规模从十亿参数增长到万亿参数，单卡显存和算力已无法满足需求，分布式训练成为大模型训练的必备能力。

**核心要解决的两大瓶颈**：

| 瓶颈 | 表现 | 解决思路 |
|------|------|----------|
| **显存瓶颈** | 175B 模型 FP16 权重约 350GB，单卡（80GB）放不下 | 模型切分到多卡 |
| **计算瓶颈** | 数万亿 token 训练需数月，单卡太慢 | 多卡并行计算 |

**常见并行策略概览**：

```
分布式训练
├── 数据并行（Data Parallelism）     → 切分数据，每张卡存完整模型
├── 模型并行（Model Parallelism）
│   ├── 张量并行（Tensor Parallelism） → 切单层参数
│   └── 流水线并行（Pipeline Parallelism） → 切模型层
└── 优化器并行（Optimizer Parallelism）→ 切分优化器状态（ZeRO）
```

## 1. 数据并行（Data Parallelism, DP/DDP）

### 1.1 核心思想

每张 GPU 都保存**完整的模型副本**，但处理不同的数据子集（batch）。反向传播后，通过集合通信同步梯度，保证所有卡的参数一致更新。

```
卡0: 数据 Batch A → Forward → Backward → 梯度 g0
卡1: 数据 Batch B → Forward → Backward → 梯度 g1
卡2: 数据 Batch C → Forward → Backward → 梯度 g2
卡3: 数据 Batch D → Forward → Backward → 梯度 g3

                ↓ All-Reduce 平均梯度
                ↓ 每张卡执行相同优化步骤
```

### 1.2 PyTorch DDP 实现

```python
import torch
import torch.nn as nn
from torch.nn.parallel import DistributedDataParallel as DDP

# 初始化进程组
torch.distributed.init_process_group(backend="nccl")

model = MyModel().to(local_rank)
# DDP 包装模型，自动处理梯度同步
ddp_model = DDP(model, device_ids=[local_rank])

# 前向/反向/优化与普通训练一致
output = ddp_model(input)
loss = criterion(output, target)
loss.backward()  # 梯度自动 All-Reduce
optimizer.step()
```

### 1.3 关键机制

- **梯度桶（Gradient Bucketing）**：将梯度按张量大小分组，小梯度合并通信，减少开销
- **重叠通信与计算**：反向传播时，当前层梯度计算完成后立即启动 All-Reduce，与上一层反向计算重叠

### 1.4 局限性

- 每张卡都存完整模型 + 优化器状态 + 梯度
- 70B 模型 FP16 训练需约 280GB 显存（模型 140GB + 梯度 140GB + 优化器状态更多），远超单卡容量
- **只适合模型能放进单卡的场景**

## 2. 张量并行（Tensor Parallelism, TP）

### 2.1 核心思想

将**单层内的参数和计算**切分到多张 GPU 上。适用于单层就太大的情况（如 FFN 中间维度 4×hidden_size）。

### 2.2 以 Transformer MLP 层为例

标准 MLP：
```python
# 输入 X: [batch, seq_len, hidden_size]
# W1: [hidden_size, 4*hidden_size]
# W2: [4*hidden_size, hidden_size]

h = X @ W1      # 线性投影，升维
h = GELU(h)     # 激活
Y = h @ W2      # 线性投影，降维
```

**列切分（Column Parallel）W1**：
```python
# W1 按列切成 2 份
卡0: W1_0 [hidden_size, 2*hidden_size]
卡1: W1_1 [hidden_size, 2*hidden_size]

h0 = X @ W1_0   # 卡0 计算一半
h1 = X @ W1_1   # 卡1 计算一半

# All-Gather 合并
h = concat([h0, h1])  # [batch, seq_len, 4*hidden_size]
```

**行切分（Row Parallel）W2**：
```python
# W2 按行切成 2 份，输入也对应切分
卡0: W2_0 [2*hidden_size, hidden_size], 输入 h0
卡1: W2_1 [2*hidden_size, hidden_size], 输入 h1

Y0 = h0 @ W2_0
Y1 = h1 @ W2_1

# All-Reduce 求和
Y = Y0 + Y1     # 每张卡得到完整结果
```

### 2.3 Attention 层的 TP

Multi-Head Attention 天然适合 TP：
```python
# 8 个 head，2 张卡
卡0: 负责 head 0-3（Q/K/V 投影矩阵对应切片）
卡1: 负责 head 4-7

# 各自计算 attention 输出
# All-Concat 合并多头结果
# 共享输出投影矩阵（行切分）
```

### 2.4 通信开销

| 切分方式 | 前向通信 | 反向通信 | 适用层 |
|----------|----------|----------|--------|
| 列切分 | All-Gather | All-Gather | 输入投影 |
| 行切分 | None（输入已切分） | All-Reduce | 输出投影 |
| Attention | All-Gather（输出合并） | All-Gather | Q/K/V/O 投影 |

**优点**：单 layer 太大时的唯一选择；通信量相对小。
**缺点**：卡间通信频繁；通常只在单机内多卡使用（NVLink 带宽高）。

## 3. 流水线并行（Pipeline Parallelism, PP）

### 3.1 核心思想

将模型**按层切分**，每张 GPU 负责连续的若干层。数据像流水线一样依次经过各阶段。

```
Batch Input
     ↓
[GPU0: Layer 1-4] ──→ [GPU1: Layer 5-8] ──→ [GPU2: Layer 9-12] ──→ [GPU3: Layer 13-16]
     ↓                      ↓                      ↓                      ↓
   中间特征              中间特征               中间特征                 输出
     ↑
[Backward 4-1] ←──── [Backward 8-5] ←──── [Backward 12-9] ←──── [Backward 16-13]
```

### 3.2 气泡问题（Pipeline Bubble）

朴素流水线的问题：GPU 大部分时间空闲等待。

```
时间 →
GPU0: F F F F F . . . . B B B B B
GPU1: . F F F F F . . . . B B B B B
GPU2: . . F F F F F . . . . B B B B B
GPU3: . . . F F F F F . . . . B B B B B

F=Forward, B=Backward, .=空闲（气泡）
```

### 3.3 优化策略

#### GPipe：微批次填充

将大 batch 切成多个 micro-batch，让流水线尽量填满：

```
GPU0: F1 F2 F3 F4 B4 B3 B2 B1
GPU1: . F1 F2 F3 F4 B4 B3 B2 B1
GPU2: . . F1 F2 F3 F4 B4 B3 B2 B1
GPU3: . . . F1 F2 F3 F4 B4 B3 B2 B1
```

#### PipeDream：1F1B（One Forward One Backward）

前向和反向交错，减少显存占用：

```
GPU0: F1 . . . B1 F5 . . . B5
GPU1: . F1 . . . B1 F5 . . . B5
GPU2: . . F1 . . . B1 F5 . . . B5
GPU3: . . . F1 . . . B1 F5 . . . B5
```

#### 交错流水线（Interleaved Pipeline）

将层交替分配到不同 GPU，进一步减少气泡：

```
# 16 层，4 张卡，每层做 2 个 stage
GPU0: Layer 1,5,9,13
GPU1: Layer 2,6,10,14
GPU2: Layer 3,7,11,15
GPU3: Layer 4,8,12,16
```

### 3.4 优缺点

**优点**：通信量少（只传层间特征）；适合跨机部署。
**缺点**：存在流水线气泡；需要保存多个 micro-batch 的中间激活（显存增加）。

## 4. 优化器并行：DeepSpeed ZeRO

ZeRO（Zero Redundancy Optimizer）是数据并行的**显存优化版本**，核心思想是**不再每张卡都存完整的优化器状态、梯度和参数**。

### 4.1 显存构成分析

训练时的显存占用：
```
显存 = 模型参数（fp16/bf16）+ 梯度（fp16）+ 优化器状态（fp32 动量等）+ 激活值

以 Adam 优化器为例（单位：字节/参数）：
- 参数：2（fp16）
- 梯度：2（fp16）
- 优化器状态：4（fp32 参数副本）+ 4（fp32 动量）+ 4（fp32 二阶矩）= 12
- 总计：16 字节/参数
```

### 4.2 ZeRO 三阶段

| 阶段 | 切分内容 | 单卡显存 | 通信量 |
|------|----------|----------|--------|
| ZeRO-1 | 优化器状态 | 原 1/4 | 与 DDP 相同 |
| ZeRO-2 | 优化器状态 + 梯度 | 原 1/8 | 1.5× DDP |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | 原 1/N（N=卡数） | 2× DDP |

#### ZeRO-1：切分优化器状态

```python
# 4 张卡，参数分为 4 份
卡0: 负责参数段0 的 Adam 状态（fp32 副本 + 动量 + 二阶矩）
卡1: 负责参数段1 的 Adam 状态
...

# 前向/反向正常进行
# 优化步骤时：
1. All-Gather fp16 梯度（如果用了 ZeRO-2/3）
2. 每张卡只更新自己负责的参数段
3. 更新后的 fp32 参数广播给所有卡
```

#### ZeRO-2：增加梯度切分

梯度也按卡分配存储，反向传播后只做 Reduce-Scatter（而不是 All-Reduce），每张卡只保留自己负责的那部分梯度。

#### ZeRO-3：参数也切分

```python
# 参数分散在每张卡上
# 前向时：需要哪层参数，就从其他卡 All-Gather 过来
# 计算完立即释放
# 反向时同样按需收集
```

### 4.3 ZeRO-Offload：CPU/NVMe 卸载

将优化器状态和计算卸载到 CPU 内存甚至 NVMe 硬盘：

```python
deepspeed_config = {
    "zero_optimization": {
        "stage": 2,
        "offload_optimizer": {
            "device": "cpu",      # 或 "nvme"
            "nvme_path": "/local/nvme"
        },
        "offload_param": {
            "device": "cpu"
        }
    }
}
```

**效果**：单卡可训练数十亿参数模型（速度较慢）。

### 4.4 配置示例

```python

    "bf16": {"enabled": true},
    "zero_optimization": {
        "stage": 3,
```

## 5. 通信原语

分布式训练依赖底层的集合通信操作，最常用的是 **NCCL**（NVIDIA Collective Communications Library）。

### 5.1 核心操作

| 操作 | 功能 | 训练中的用途 |
|------|------|--------------|
| **Broadcast** | 一张卡数据发给所有卡 | 初始化参数同步 |
| **All-Reduce** | 所有卡数据求和/平均，结果散布到所有卡 | DDP 梯度同步 |
| **All-Gather** | 每张卡提供部分数据，汇总后所有卡拥有完整结果 | 张量并行合并输出 |
| **Reduce-Scatter** | 先求和，再按卡切分结果 | ZeRO-2/3 梯度/参数分发 |
| **All-to-All** | 每张卡发送不同数据给不同卡 | MoE 专家并行路由 |

### 5.2 通信拓扑

```
单机多卡（DGX A100）:
NVLink 全互联，带宽 600GB/s
├── 适合：张量并行（TP），通信频繁

多机多卡:
InfiniBand / RDMA 网络
├── 机间带宽 100-400Gbps（远低于 NVLink）
├── 适合：数据并行、流水线并行（通信少）
└── 避免：跨机张量并行
```

## 6. 实际训练中的混合并行

大规模模型训练不会只用一种并行，而是组合多种策略形成**多维并行**。

### 6.1 并行维度组合

```
总 GPU 数 = 数据并行度 (DP) × 张量并行度 (TP) × 流水线并行度 (PP)

示例：GPT-3 175B 训练
- 模型层数 96，隐藏维度 12288
- TP = 8（单机内 NVLink，切 hidden_dim）
- PP = 12（跨机，切 96 层为 12 个 stage）
- DP = 16（不同节点组处理不同数据）
- 总 GPU = 8 × 12 × 16 = 1536 张 A100
```

### 6.2 各并行策略的职责

| 并行 | 解决什么问题 | 通常部署范围 |
|------|--------------|--------------|
| **TP** | 单层太大（如 hidden=12288） | 单机内（NVLink） |
| **PP** | 模型层数多 | 跨机，同机架 |
| **DP** | 加速训练、扩大 batch size | 跨机架、跨集群 |
| **ZeRO** | 单卡放不下优化器状态 | 替代或补充 DP |

### 6.3 主流框架

| 框架 | 核心优势 | 适用场景 |
|------|----------|----------|
| **Megatron-LM** | TP + PP 极致优化，NVIDIA 官方 | 超大模型预训练（GPT/T5） |
| **DeepSpeed** | ZeRO + 3D 并行，集成方便 | 微调、显存受限场景 |
| **FSDP** | PyTorch 原生，API 简洁 | 中小规模，易迁移 |
| **Colossal-AI** | 统一封装，国产 | 多种并行一键切换 |
| **vLLM / Ray** | 推理侧分布式 | 高并发服务部署 |

## 7. 小结与选型思路

```
模型是否能放进单卡？
├── 能 → 直接用 DDP / FSDP（最简单）
└── 不能 → 需要模型并行
    ├── 单层太大（如 FFN 升维后 > 显存）→ 张量并行（TP）
    ├── 层数太多（如 100+ 层）→ 流水线并行（PP）
    └── 优化器状态占满显存 → ZeRO-2/3

跨机部署？
├── 同机 NVLink → TP 优先
└── 跨机网络 → PP / DP 优先，避免跨机 TP
```
